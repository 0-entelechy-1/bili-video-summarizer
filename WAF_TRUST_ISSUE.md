# B站 WAF 信任机制与 SESSDATA 412 问题排查报告

> 适用对象：未来接手本项目的 AI / 开发者
> 编写时间：2026-06-07
> 触发问题：项目改造为使用 yt-dlp 下载字幕后，扫码登录的新 SESSDATA 始终无法调通 `/x/player/wbi/v2`（HTTP 412），而本地现存的 `outputs/videos/cookies.txt` 同等条件下完全可用

---

## 1. TL;DR（30 秒结论）

| 现象 | 结论 |
|------|------|
| `--login` 重新登录后跑程序 → 字幕下载 412 | **正常**——新 SESSDATA 在 B站 WAF 黑名单里 |
| 手动终端用 `outputs/videos/cookies.txt` 跑 yt-dlp → 成功 | **正常**——该 cookies 是 yt-dlp 自己覆盖写入的，含真实浏览器指纹 |
| 程序走 `config.transcriber.cookies_file` → 成功 | **正常**——绕开 QR 登录路径 |
| 怎么让新 SESSDATA 也能用 | **用 `config.transcriber.cookies_file` 指向已通过 yt-dlp 验证的 cookies 文件** |

**核心机制**：B站 WAF（基于 `openresty`）对 `/x/player/wbi/v2` 端点执行"会话信誉评分"——`SESSDATA + buvid3 + b_nut` 组合在 B站风控系统里有"信任分"，新扫码登录产生的 SESSDATA 默认 0 分，访问 `wbi/v2` 必返 412。该评分由 B站服务端控制，**无法用代码绕过**。

## 1.5 什么是 buvid3 / b_nut？（常被误认为是浏览器 cookie）

**澄清**：`buvid3` 和 `b_nut` **不是浏览器 cookie**，是 B站服务端在客户端**第一次访问 B站域名时通过 `Set-Cookie` 头下发的风控标识**。

| 字段 | 实际含义 | 长度 / 格式 | 谁生成的 | 谁决定其值 |
|------|---------|-----------|---------|----------|
| `buvid3` | 浏览器唯一标识（"Browser Unique Visitor ID v3"）。B站用来关联"这个客户端的访问历史、风控档案、广告画像"。 | UUID 格式（4 段：`8-4-4-4-12` 字符）+ 后缀（`infoc` / `a` 等） | B站服务端 | B站算法（与 IP、User-Agent、TLS 指纹、Canvas 指纹等多维信息关联生成） |
| `b_nut` | "上次访问时间戳"（"Browser NuT"，与 buvid3 配对）。每次 B站认为"该 buvid3 又访问了一次"，就会更新这个值。 | 10 位 Unix 时间戳（如 `1780820548` = 2026-06-07 16:22:28） | B站服务端 | B站服务端（每次 Set-Cookie 时刷新） |

**"infoc" 后缀**（如本项目 cookies.txt 里的 `A71CF0C3-...-CDE9A48964infoc`）表示这个 buvid3 是通过 B站**前端信息指纹**流程（`/info` JS 收集 canvas、UA、TLS 等信息后调用 B站接口）下发的，对应 B站风控里"较可信的客户端"分类。如果是 "a" 后缀则是 API 流程下发的，可信度更低。

**为什么 `requests` 库会"自动"拿到这两个 cookie？**——因为 `requests.Session` 在收到 HTTP 响应时会**自动**把 `Set-Cookie` 头里的 cookie 存进 `session.cookies`。本项目调用：

```python
# src/bili_analyzer/api/auth.py
session.get("https://passport.bilibili.com/x/passport-login/web/qrcode/generate")
# ↑ 这一步 B站就会在响应里下发 buvid3 + b_nut
```

B站就是通过这种方式给所有"第一次来的"客户端打上唯一标识的——浏览器会存进自己的 cookie jar，Python 的 `requests` 库会存进 `session.cookies`，任何 HTTP 客户端都会得到一份。

**WAF 为什么"信任"这一个 buvid3？**——`buvid3` 本身只是个 UUID，本身不携带信任信息。WAF 的"信任分"是基于"`buvid3` + 该客户端 IP + 该客户端 TLS 指纹"组合在 B站访问历史中的"健康度"计算的。本项目工作 cookies.txt 里的这个 `A71CF0C3-...infoc` 之所以被 WAF 放行，是因为它**通过 yt-dlp 的多次成功访问**（每次 yt-dlp 调 B站都会带上它）已经"养"了一段时间的信任；新 SESSDATA 配套的那个新 buvid3 还没"养"出信任。

## 2. 什么是 WAF？

**WAF** = Web Application Firewall（Web 应用防火墙）。本项目面对的 WAF 是 B站自研、基于开源 `openresty`（nginx + Lua）的反爬系统。

它的拦截响应特征：
- HTTP 状态码：**412 Precondition Failed**
- `Server: openresty` 响应头
- 响应体是一段 **HTML**（不是 JSON），内容是 B站统一的"访问频次过高"提示页

WAF 在网络层（CDN 边缘）就拦下了请求，根本到不了 B站后端应用，所以 Python 代码无论怎么改 `requests` 库调用，都会被它拒绝。

WAF 的判定维度（推测）：

| 维度 | 说明 | 是否可绕过 |
|------|------|------------|
| SESSDATA 信誉 | 该 SESSDATA 在 B站系统里有没有"使用历史" | ❌ 不可绕过 |
| buvid3 浏览器指纹 | buvid3 是 B站下发的浏览器唯一标识，关联到"这台浏览器" | ❌ 不可绕过（需要真实浏览器） |
| b_nut 时效 | b_nut 是"上次访问时间"戳，太新会被认为可疑 | ⚠️ 理论上可调，实际无法伪造 |
| TLS 指纹 (JA3/JA4) | `requests` / `urllib3` 的 TLS 握手特征与真实浏览器不同 | ❌ 不可绕过（需要 curl_cffi / tls-client 之类） |
| HTTP/2 帧特征 | 浏览器和 Python 客户端的帧时序不同 | ❌ 不可绕过 |
| IP 信誉 | IP 是否被标记为数据中心、代理 | ⚠️ 取决于网络环境 |
| 请求速率 | 单位时间请求数 | ✅ 可控（加 sleep） |

**本项目里的 412 = WAF 命中 = 上面若干维度的组合判定失败**。

## 3. 排查时间线（关键发现链）

### 第一阶段：复现 412
- 视频下载 ✓，字幕下载 ✗ → 412
- 错误信息：`ERROR: [BiliBili] 1Rr5v6wE1F: Unable to download JSON metadata: HTTP Error 412`

### 第二阶段：定位是哪个端点
用 `requests` 直接打几个 B站 API：

| 端点 | 新 SESSDATA | 工作 cookies |
|------|-------------|--------------|
| `/x/web-interface/view` (基础信息) | 200 ✓ | 200 ✓ |
| `/x/player/v2` (老版播放) | 200 ✓ | 200 ✓ |
| **`/x/player/wbi/v2`** (新版取字幕) | **412 ✗** | **200 ✓** |

→ 锁定是 **`/x/player/wbi/v2`** 端点。该端点是 yt-dlp 取字幕元数据必经的，yt-dlp 源码在 [extractor/bilibili.py:198](https://github.com/yt-dlp/yt-dlp/blob/master/yt_dlp/extractor/bilibili.py)：

```python
'https://api.bilibili.com/x/player/wbi/v2', video_id,
query={'bvid': video_id, 'cid': cid},
note=f'Extracting subtitle info {cid}'
```

### 第三阶段：排除"buvid3 缺失"的锅
最初怀疑新 SESSDATA 没有 `buvid3`，于是加 `_warm_buvid3()` 预热逻辑（访问 `https://www.bilibili.com` 拿 buvid3 + b_nut）。

| 场景 | 状态 |
|------|------|
| 新 cookies 8 项（无 buvid3） | 412 ✗ |
| 新 cookies + 预热 buvid3（10 项） | 412 ✗ |
| 新 cookies + 工作的 buvid3 + 工作的 b_nut（10 项） | 412 ✗ |
| 工作 cookies 10 项 | 200 ✓ |

→ **加 buvid3 反而没救**，因为 `_warm_buvid3` 用 `requests` 库访问 B站首页拿到的 buvid3 是"低信任"的，WAF 反而更怀疑。**已从代码中删除 `_warm_buvid3` 调用**（函数保留作为历史参考），见 [src/bili_analyzer/downloader/ytdlp.py](file:///c:/实用/视频内容总结/bili_analyzer/src/bili_analyzer/downloader/ytdlp.py)。

### 第四阶段：排除"IP 限速"
担心是请求速率触发，等 30 秒再测：

| 测试 | 状态 |
|------|------|
| 等待 30 秒后用工作 cookies | 200 ✓ |
| 等待 30 秒后用新 cookies | 412 ✗ |
| 再等 5 秒用工作 cookies | 200 ✓ |

→ **不是 IP 限速**，是 SESSDATA 维度的"白/黑"判断。

### 第五阶段：破解"都是今天登录的"疑问
**用户原话**："`outputs/videos/cookies.txt` 其实也是我今天才刚刚登录的，只比 `--login` 重新登录早了十几分钟"

实际查证：
- 文件 `LastWriteTime` = `2026-06-07 20:08:32`（今天）—— 看似是今天生成的
- 但文件头写着 **`# This file is generated by yt-dlp. Do not edit.`** —— **该文件不是 `outputs/videos/bili_login.py` 写的，是 yt-dlp 覆盖写入的**
- 文件里的 `b_nut=1780820548` 对应 **2026-06-07 16:22:28 北京时间**（今天下午 4 点多，约 4 小时前）
- 文件里的 `buvid3=A71CF0C3-2EE1-B16C-47CE-9459238CDE9A48964infoc` 是 UUID 格式 + `infoc` 后缀

**真相（结合 2026-06-08 03:00 的进一步排查更新）**：

```
历史：
  用户在 Edge / 夸克 / Tabbit 等浏览器中登录过 B站
  → 各浏览器持久化各自的 SESSDATA / buvid3 / b_nut 等
  → 但这与本 cookies.txt 无关（见下文验证）

2026-06-07 16:22 左右（b_nut 时间戳 = 1780820548）：
  程序的 bili_login.py 调 https://passport.bilibili.com/x/passport-login/web/qrcode/generate
  → B站响应 Set-Cookie: buvid3=A71CF0C3-...infoc; b_nut=1780820548
  → requests.Session 自动把这两项 cookie 存进 session.cookies
  → 这是 buvid3 / b_nut 第一次出现的时刻

2026-06-07 18:45 左右：
  用户扫码完成，bili_login.py 调 poll_login_status 拿回 SESSDATA / bili_jct 等
  → 此时 session.cookies 已含 10 项（包含 buvid3 + b_nut）
  → bili_login.py 把 10 项写进 cookies.txt

2026-06-07 18:57 ~ 20:08（多次）：
  用户用 cookies.txt 调 yt-dlp 下载视频 / 字幕
  → yt-dlp 用文件头 "# This file is generated by yt-dlp" 覆盖写回 cookies.txt
  → 但 buvid3 / b_nut 没变（除非 B站重新下发）

2026-06-07 20:08 之后：
  cookies.txt 最终内容：8 项（看 lastWriteTime 后的内容）
  → 仍是原 buvid3 A71CF0C3-... + b_nut 1780820548
```

**关键验证**（2026-06-08 03:00）：用 `scan_buvid3_source.py` 扫描用户电脑上的所有浏览器 cookies 数据库：

| 浏览器 | 路径 | buvid3 匹配工作文件? |
|--------|------|---------------------|
| B站 PC 客户端 | `C:\Users\29146\AppData\Roaming\bilibili\Network\Cookies` | ✗ 数据库里**根本没有** buvid3 这一项 |
| Microsoft Edge | `C:\Users\29146\AppData\Local\Microsoft\Edge\User Data\Default\Network\Cookies` | ✗ 文件被 Edge 进程占用，无法访问（提示：需要先关 Edge） |
| 夸克浏览器 | `C:\Users\29146\AppData\Local\Quark\User Data\Default\Network\Cookies` | ✗ 自己的 buvid3 是 `65151600-3A83-...`，**与工作文件的 A71CF0C3-... 不同** |
| Tabbit Browser | `C:\Users\29146\AppData\Local\Tabbit Browser\User Data\Default\Network\Cookies` | ✗ 自己的 buvid3 是 `626A2F43-D2D2-...`，**与工作文件的 A71CF0C3-... 不同** |

**结论**：工作 cookies.txt 的 `buvid3` 和 `b_nut` **不是来自用户电脑上的任何浏览器**，也不是 bili_login.py 自己生成的。它们是 B站服务器在 2026-06-07 16:22 左右，**响应程序的 `requests` 请求时通过 Set-Cookie 头下发的**。

**那为什么这一个 buvid3 就能"养"出 WAF 信任，--login 重新登录后的新 SESSDATA 就不行？**——见第 4 节。

## 4. "养"出 SESSDATA 信任的可行方法

**结论先行**：B站 WAF 的信任评分机制不公开，**目前没有可靠的方法能从脚本里让新 SESSDATA 快速获得 wbi/v2 信任**。下面是一些**可能有效**的方法，按可行性排序：

### 4.1 直接用现成的 cookies 文件（推荐，**最稳**）
项目 `config.yaml` 里已经支持 `transcriber.cookies_file` 配置项：
```yaml
transcriber:
  cookies_file: C:/实用/视频内容总结/bili_analyzer/outputs/videos/cookies.txt
```
**优点**：开箱即用，不依赖 WAF 评分
**缺点**：需要用户手动从浏览器导出 / 维护

### 4.2 用浏览器扩展（如 "Get cookies.txt LOCALLY"）导出浏览器 cookies
浏览器里的 cookies 是 WAF 最信任的（因为有真实的访问历史）。导出流程：
1. Chrome / Edge 装 "Get cookies.txt LOCALLY" 扩展
2. 登录 B站后，在 B站页面点扩展 → Export
3. 把 Netscape 格式的 cookies.txt 保存到任意路径
4. 配置 `transcriber.cookies_file` 指向这个文件

**优点**：WAF 一定放行
**缺点**：每次 SESSDATA 过期（~2-3 个月）需要重新导出

### 4.3 让 SESSDATA 在浏览器里"过几天"
**理论可行**：
1. 浏览器登录 B站，保留登录态
2. 几天/几周内正常浏览 B站
3. WAF 累积该 SESSDATA + 该 IP + 该浏览器指纹的信任分
4. 几天后用 `outputs/videos/bili_login.py` 重新生成 cookies.txt
5. 这次保存的 cookies 应该能通过 WAF（因为新的 SESSDATA 是从浏览器持久化继承的）

**优点**：不需要用户改架构
**缺点**：
- 等待时间不可控（几天到几周不等）
- 等待期间新 SESSDATA 仍然会被 wbi/v2 拒绝
- 没有文档证据表明这个机制确实有效，**只是基于 WAF 通用机制的推测**

### 4.4 用真实浏览器自动化（playwright / selenium）
理论上：让程序通过真实浏览器跑 QR 登录 → 自动浏览 B站几分钟 → 提取 cookies

**优点**：能模拟真实用户
**缺点**：
- 改动巨大（项目要从 `requests` 切换到 playwright）
- 启动速度慢、依赖重
- B站仍然可能识别为自动化
- **当前不推荐**

### 4.5 用 cookies 池（多个 SESSDATA 轮换）
不可行：每次都得到 0 信誉分的新 SESSDATA，问题不变。

### 4.6 等待 B站官方降低 WAF 严格度
不可行：WAF 是核心反爬基础设施，只会越来越严。
